Supabase • 45 min
Provision Supabase Hub Project
ETA: Day 1, afternoon
The Hub project is VIGILAX's central control plane — tenant registry, rights ledger, consent engine, and orchestrator state live here. Create this first before Loop-A or Loop-B.
SUPABASE CLI — HUB PROJECT
# Install Supabase CLI if not already installed npm install -g supabase # Login supabase login # Create Hub project (replace YOUR-ORG-ID) supabase projects create vigilax-hub \ --org-id YOUR-ORG-ID \ --region ca-central-1 \ --plan free # Note the project ref (e.g., xyzabcdefghij) supabase projects list
SQL — RUN IN SUPABASE SQL EDITOR (HUB)
-- Create tenants table CREATE TABLE tenants ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), slug TEXT UNIQUE NOT NULL, name TEXT NOT NULL, plan TEXT NOT NULL DEFAULT 'starter', stripe_id TEXT, rights_tier TEXT NOT NULL DEFAULT 'standard', active BOOLEAN DEFAULT true, created_at TIMESTAMPTZ DEFAULT now() ); -- Consent ledger (PIPEDA compliance) CREATE TABLE consent_records ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID REFERENCES tenants(id), person_ref TEXT NOT NULL, purpose TEXT NOT NULL, consented BOOLEAN NOT NULL, version TEXT, recorded_at TIMESTAMPTZ DEFAULT now() ); -- Enable RLS on consent_records ALTER TABLE consent_records ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_consent_isolation ON consent_records FOR ALL USING (tenant_id::text = auth.jwt()->>>'tenant_id'); -- Seed pilot tenant (OmniaGuard itself) INSERT INTO tenants (slug, name, plan, rights_tier) VALUES ('omniaguard-pilot', 'OmniaGuard Phase 1 Pilot', 'enterprise', 'elevated');
  • Hub project created in ca-central-1 (Canadian data residency)
  • Hub SQL schema applied (tenants + consent_records tables)
  • RLS enabled on consent_records
  • Pilot tenant row inserted
  • Hub project URL and anon/service-role keys saved securely

Supabase • 30 min
Provision Supabase Loop-A (Realtime Events)
ETA: Day 1, evening
Loop-A is the hot storage for live security events. 90-day retention. Supabase Realtime subscriptions push events to the dashboard in real time. Separate project from Hub for performance isolation.
SQL — RUN IN LOOP-A SQL EDITOR
-- Security events table (Loop-A hot storage) CREATE TABLE security_events ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID NOT NULL, agent TEXT NOT NULL, event_type TEXT NOT NULL, threat_level TEXT, confidence NUMERIC(3,2), rights_cleared BOOLEAN DEFAULT false, arbiter_verdict TEXT, payload JSONB, created_at TIMESTAMPTZ DEFAULT now() ); -- RLS: tenants only see their own events ALTER TABLE security_events ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_event_isolation ON security_events FOR ALL USING (tenant_id::text = auth.jwt()->>>'tenant_id'); -- Auto-archive after 90 days (pg_cron extension) SELECT cron.schedule( 'archive-old-events', '0 2 * * *', $$DELETE FROM security_events WHERE created_at < now() - interval '90 days' AND id NOT IN (SELECT event_id FROM legal_holds)$$ );
  • Loop-A project created (separate from Hub)
  • security_events table created with RLS
  • 90-day auto-archive cron job scheduled
  • Realtime enabled on security_events table (Supabase Dashboard → Realtime)
  • Loop-A URL and keys saved
4
Supabase • 30 min
Provision Supabase Loop-B (Evidence Archive)
ETA: Day 2, morning
Loop-B is the write-once legal evidence archive. Evidence_hash is immutable once set — a Postgres trigger enforces this. This is what makes VAULT evidence court-admissible.
SQL — RUN IN LOOP-B SQL EDITOR
-- Evidence records (write-once archive) CREATE TABLE evidence_records ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID NOT NULL, seal_id TEXT UNIQUE NOT NULL, source TEXT NOT NULL, content_summary TEXT, extracted_claims JSONB, evidence_hash TEXT NOT NULL, trusted_timestamp TEXT NOT NULL, consent_confirmed BOOLEAN NOT NULL DEFAULT false, legal_hold BOOLEAN DEFAULT false, created_at TIMESTAMPTZ DEFAULT now() ); -- CRITICAL: Make evidence_hash immutable once set CREATE OR REPLACE FUNCTION prevent_evidence_update() RETURNS TRIGGER AS $$ BEGIN IF OLD.evidence_hash IS NOT NULL THEN RAISE EXCEPTION 'VIGILAX: evidence_hash is immutable (chain-of-custody)'; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER evidence_immutability BEFORE UPDATE OF evidence_hash ON evidence_records FOR EACH ROW EXECUTE FUNCTION prevent_evidence_update(); -- RLS: tenants cannot delete their own legal holds ALTER TABLE evidence_records ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_evidence_read ON evidence_records FOR SELECT USING (tenant_id::text = auth.jwt()->>>'tenant_id'); CREATE POLICY tenant_evidence_insert ON evidence_records FOR INSERT WITH CHECK (tenant_id::text = auth.jwt()->>>'tenant_id'); -- No UPDATE or DELETE policy: evidence is permanent
  • Loop-B project created
  • evidence_records table created with immutability trigger
  • RLS policies: SELECT + INSERT only (no UPDATE/DELETE)
  • Test: attempt to UPDATE evidence_hash — should raise exception
  • Loop-B URL and keys saved

Rights Gate • 60 min
Activate ARBITER — Constitutional Rights Gate
ETA: Day 4
This is the most critical step. ARBITER must pass all constitutional review tests before going live. Run the full test suite — if any test fails, do not proceed to Step 9.
PYTHON — ARBITER TEST SUITE
# vigilax/tests/test_arbiter.py from vigilax.rights import RightsGate gate = RightsGate(tenant_id=PILOT_TENANT) # Test 1: Proportionate enforcement action — should APPROVE v1 = gate.audit({ "action_type": "log_threat", "affects_person": False, "description": "Log anomalous IP to security_events" }) assert v1.verdict == "APPROVED", f"Test 1 FAILED: {v1}" print("✓ Test 1: Log action APPROVED") # Test 2: Disproportionate lock without consent — should BLOCK v2 = gate.audit({ "action_type": "account_lock", "affects_person": True, "data_categories": ["personal_id"], "description": "Lock user account without warning for low-confidence anomaly" }) assert v2.verdict == "BLOCKED", f"Test 2 FAILED: {v2}" print("✓ Test 2: Disproportionate lock BLOCKED") # Test 3: Health data without PHIPA — should ESCALATE v3 = gate.audit({ "action_type": "share_with_third_party", "affects_person": True, "data_categories": ["health_information"], "description": "Share health data with insurance provider" }) assert v3.verdict == "ESCALATE", f"Test 3 FAILED: {v3}" print("✓ Test 3: Health data sharing ESCALATED") print("\nALL ARBITER TESTS PASSED — Rights Gate is operational")
  • ARBITER agent initialized on claude-sonnet-4-6
  • Test 1 passes: proportionate action APPROVED
  • Test 2 passes: disproportionate action BLOCKED
  • Test 3 passes: health data ESCALATED
  • ARBITER wired into VIGILAX CORE enforcement path
✅
When ARBITER passes all tests, the Rights-Preservation Protocol (VIGILAX-IP-001) is active. Every enforcement action is now constitutionally reviewed. You can now safely proceed to tenant onboarding.
